# Engineering Log

How Foundry took shape: the design goals, the phases, and the decisions that shaped the build. See [NOTES.md](NOTES.md) for the polished architectural reference and [FRICTION.md](FRICTION.md) for points of friction during the build.

## Summary

Foundry is a suite of Claude Code skills for software delivery integrity: five promises between an idea and its release (intent declared and audited, execution context preserved, the outcome instrumented, docs aligned with reality, readiness audited before real users), each kept by a skill. The flagship, production-audit, audits a whole application across six core dimensions plus the conditional ones a project's surface implies, and ends in a scope-qualified ship/no-ship verdict against a versioned JSON report contract. It began as a single audit skill built to fill a gap (a security review covers security, a diff review covers a diff; nothing audited a complete application and returned a release decision), grew into the nine-skill suite, and ships with a Fly-deployed site where reports render behind an email gate.

### Phase 1: Skill + report contract (2026-07-10)

Wrote `skills/production-audit/SKILL.md` as an original audit method. Its distinguishing choices: a flow-driven method instead of category checklists, required artifacts (authorization matrix, failure-path table), a defined severity rubric, evidence-class confidence, a false-positive kill list, an adversarial verification phase, and an honest static/runtime boundary. ### Phase 2: First full run against a real application (2026-07-10)

Ran the skill full-depth, static-only, against a real Next.js + Postgres application (~29k LOC, Next 16 + Supabase). Six parallel dimension agents produced 38 findings; three adversarial verifiers refuted the high finding and the verdict-driving mediums. Outcome: zero critical, zero high (the security-headers finding was downgraded high to medium for the invite-only dogfood blast radius), 7 medium, 20 low, 11 informational. Verdict: ship with known risks. The security dimension came back clean (full authorization matrix verified, no permissive RLS write policy, defended prompt-injection surface). The report was published to the gated `/reports` section. The run validated the skill end to end with no refactor needed; it is the case study for the site.

### Phase 8: document becomes idempotent (2026-07-11)

Reframed `document` as an idempotent reconcile operation in response to "it should be idempotent, I should be able to run it as frequently as I want to make sure all the documentation surfaces are up to date; the first time I run it is to inform I want documentation rigor." The skill now detects state (establish on first run, reconcile on every run after), holds a convergence test (a second unchanged run produces no diff), and enumerates the surfaces a re-run checks (the drift-prone deck and showcase copy; the doc-set tabs are inherently current). Framed its relationship to doc-drift (docs-versus-code) as document owning the presentation layer (deck, hub, showcase copy) versus shipped capability. Reconciled document's own one-liners in README and PRODUCT to match. The manual coverage sweep that preceded it is what a document re-run now automates.

### Phase 12.10: Positioning copy, tune-vs-fixed and outer-loop (2026-07-14)

A colleague evaluating the suite raised two things that were really one: "can I tune what is production ready, or is it your own recipe?" and "have you seen Matt Pocock's work?" Both ask whether Foundry is a personal idiolect or connects to the shared practice, and the honest read was that the answers existed in the product but nowhere on its surfaces. Two framings were written into PRODUCT.md as the source of truth and echoed across the homepage, the skills directory, the deck, the README, and the production-audit skill card. First, what you tune versus what is fixed: you own the scope (the release policy, which gates apply, the risk tier, the risks you accept on the record, and dimensions that resolve by applicability), and you do not own the bar (the severity rubric and the meaning of the verdict are fixed on purpose, because a standard you can lower until everything reads green is the false comfort a second read exists to replace). Foundry is opinionated where opinion is the product, and owning that is a stronger answer than pretending everything is configurable. Second, where Foundry fits: it is the outer loop, the release gate, composing with the inner-loop skills a builder already uses rather than replacing them, through the same install channel, which positions it as a good citizen of the emerging skills ecosystem rather than a rival to it. A risk-tier-moves-the-verdict-tolerance extension was left as an open post-launch decision, not built. Copy only; deployed to the live site so more eyes could read it before the Phase 13 flip.

## Origin and direction

This section records the decisions that shaped the build, in the project's own voice.

- **The brief.** Design a whole-application audit that covers security, concurrency, reliability, accessibility, UI consistency, and infra; that traces flows end to end rather than reviewing files in isolation; that grades every finding by severity and confidence with evidence; and that ends in a release recommendation, safe to ship, ship with known risks, or do not ship. Make it a reusable skill, not a one-off, and make it stack-agnostic.

- **An original method, not a checklist.** The naive shape of an audit is category-driven: flat lists of things to look for, self-graded confidence, and browser-dependent findings asserted from static code. The skill was authored to reject that shape. It works flow by flow, produces artifacts a reviewer can finish (an authorization matrix, a failure-path table), earns severity against a rubric, ties confidence to evidence classes, forbids the predictable false positives with a kill list, refutes its own critical and high findings, and states plainly what it did not look at.

- **A full product, not a skill file.** The work is delivered in the standard shape: a private repo, the full doc set (README, CLAUDE, PRODUCT, DESIGN, PROMPTS, NOTES, FRICTION, TODOS), a versioned JSON report contract, a marketing site, a Behind-the-Build hub, and an overview deck.

- **Published, gated reports.** Rather than maintaining markdown reports and translating them to HTML, the skill emits JSON against a schema and the site renders every report through one template. Reports carry vulnerability findings for live products, so they are published to a `/reports` section behind a server-side email gate.

- **Validated against a real product.** The skill's first run audited a real Next.js + Postgres application, a near-production product, end to end. That run is the case study.
