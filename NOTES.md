## The Goal

Production Audit is a pre-launch gate for AI-built products: a Claude Code skill that audits a whole application across security, concurrency, reliability, accessibility, UI consistency, and infra, then delivers a ship/no-ship verdict backed by verified evidence. It exists because the toolset had security-only, diff-only, and frontend-only reviews, but nothing that answers the question a builder actually asks before inviting users: is this safe to ship? It serves the person shipping an AI-built product who wants an honest read before real users arrive, and it is built with an open-source posture.

## Architectural Decisions

### 2026-07-10: JSON-native reports with a versioned schema, no markdown step

The skill emits reports as JSON conforming to `schema/audit-report.schema.json`, and the site renders them through one shared template. Reasoning: maintaining markdown reports and translating them to HTML is a drift-prone double representation; a schema makes every report render consistently and makes the contract testable. Rejected alternative: markdown reports in `tmp/` rendered via the MarkdownDoc pattern.

### 2026-07-10: Site is Next standalone, not static export

A static export plus nginx was the initial choice since the site has no server logic. Rejected once `/reports` was added: reports contain vulnerability findings for live products, and an email gate on a static site can only be client-side, which is bypassable. The gate must be checked server-side, so the site uses Next's `output: "standalone"` node deploy. Docs and deck pages remain effectively static inside the standalone app.

### 2026-07-10: Skill installed user-global via symlink, canonical copy in the repo

`~/.claude/skills/production-audit` is a symlink to this repo's `skills/production-audit/` directory. Reasoning: the skill must be versioned, reviewed, and shipped like product code, while staying available from any project. Rejected alternatives: a copy in `~/.claude/skills` (diverges silently) or a project-local skill (invisible from other projects, which is the whole point).

### 2026-07-10: Server-side gate, HMAC cookie, no auth system

The `/reports` gate verifies a signed HTTP-only cookie in the `/reports` layout (Node runtime, node:crypto), not in middleware. Reasoning: reports carry vulnerability findings for live products, so the check must be server-side and uncacheable (`force-dynamic`); a client gate on otherwise-static pages is bypassable. The cookie is an HMAC over the email + issued-at; verification re-checks the allowlist, so removing an email revokes outstanding cookies. Rejected alternatives: a full auth system (overkill for one reviewer), Supabase magic links (a DB dependency for a docs site), and an edge-middleware gate (Web Crypto works but node:crypto in a server component is simpler and the layout already wraps every /reports route). The unlock form lives at top-level `/unlock`, outside `/reports`, so the guarded layout can redirect to it without a loop.

### 2026-07-10: Dogfood the audit's own findings in the site

next.config.ts ships the full security-header set (CSP, X-Frame-Options, HSTS, nosniff, Referrer-Policy, Permissions-Policy) that the first audit report flagged as missing (INF-01). An audit product that shipped without its own headline recommendation would undercut the whole premise. CSP ships enforcing (not report-only) because the site is fully self-hosted with no third-party scripts and next/font self-hosts fonts, so a strict policy breaks nothing here.


### 2026-07-11: Rebrand to Foundry, a Horizon Foundry product

The repo and site rebranded from a standalone "Production Audit" product into Foundry, the skills suite, a product of the Horizon Foundry studio. Adopted the Horizon Foundry identity (steel #14191F / Brand Slate #334455 ground, amber #D99A2E horizon accent, the H/F mark, Inter display) and kept IBM Plex Mono plus the severity palette for the audit/report instrument layer, so the tool keeps its forensic register inside the brand. Production Audit is the flagship skill. Rejected: leading the hero with the abstract "Distill. Forge. Deliver." triad (it reads as externally abstract), kept it as a supporting motif and led concrete instead. The site is the Foundry skills product, not the Horizon Foundry corporate homepage (separate, and will link here).

## Product Decisions

### 2026-07-10: An original method, not a category checklist

The audit method is authored, not assembled from a checklist. The naive shape of an audit is category-driven: flat lists of things to look for, self-graded confidence, and browser-dependent findings asserted from static code. This skill rejects that shape deliberately. Category-driven checklists become a flow-driven method with required artifacts (authorization matrix, failure-path table); undefined severity levels get an exploitability-times-impact rubric; self-graded confidence becomes evidence classes; browser-dependent claims get an honest static/runtime boundary instead of fabricated assertions; and a false-positive kill list encodes the known ways an AI audit embarrasses itself. The rationale is that a finding is only worth reading if it can be finished, earned, and refuted.

### 2026-07-10: Reports are gated; public surfaces get aggregates only

The `/reports` section is the product's review interface for the maintainer, gated by allowlisted email + signed cookie (a private-review pattern adapted for a site with no auth system). Ungated pages may show process, verdict levels, and aggregate counts, never exploitable specifics, because the audited application (a real Next.js + Postgres app) is live.

### 2026-07-10: Verify verdict-drivers, not just Critical/High

The skill mandates adversarial verification of Critical/High findings. In the first full run there were no survivors at those tiers (the one high, INF-01, was downgraded), yet the verdict still turned on four medium findings. Those mediums were verified too, because a "ship with known risks" verdict is only as trustworthy as the findings it accepts as known risks. The lesson folds back conceptually: the verification target is "findings that drive the verdict," which is a superset of Critical/High. The skill text says verify Critical/High always; the judgment to also verify verdict-blocking mediums is worth keeping. It is not a skill defect, so no REFACTOR this run.

### 2026-07-11: Renamed behind-the-build to document

The `behind-the-build` skill became `document`. Reasoning: for a suite meant to be shared publicly, the skill names are the interface, and "behind-the-build" names an artifact (the hub) rather than an action, so it never reads as a step you would invoke. It has to be asked for by name every time. "document" reads as a lifecycle stage (scaffold, build, document, audit, ship), which is how it stops being a special request. A concrete signal: a behind-the-build had to be instructed by name each time, which is the naming problem made concrete. The hub the skill produces is still called "Behind the Build"; the skill (the action) is `document`. Also wired `scaffold` to hand off to `document` at project start (step 6) so the hub stands up early and never becomes a later, already-drifted addition. Rejected: folding documentation into `behind-the-build` as-is (an earlier design recommended against a separate `/document` skill and made behind-the-build own deck authoring; that was overridden for the sharing/discoverability reason above, which is correct).

### 2026-07-11: foundry as a conductor over the pre-ship gates, not a run-everything

Added a `foundry` skill: one command to run the standing pre-ship gates. Reasoning: the disciplines are easy to skip one at a time (nobody is blocked without them), which is the same friction that drove the `document` rename, so a single entry point that runs the sweep removes the remembering. Chose the scope deliberately (a "pre-ship readiness pass" over a full lifecycle conductor): it runs only the four *standing* gates, document, mobile, instrumentation, production-audit, in that order (document first so later gates read current docs; the two experience gates before the verdict so the audit ratifies them; production-audit last because it is the verdict and already treats mobile+instrumentation as caps). It deliberately excludes the once-only/session skills (scaffold, resume, phase-plan, commit) and continuous disciplines (doc-drift, brand-voice), because those are not a linear pass; when the project is not at a release decision, foundry names the right earlier skill and stops. Key constraint, from the suite's own invocation rule: foundry is a **conductor, not a doer**, it invokes each real skill (honoring that skill's gate) and never reproduces the work inline, and it synthesizes on top of the audit's JSON contract rather than inventing a second report format. Rejected: full lifecycle auto-detection (more magic, more ways to misfire on a wrong state guess); a leaner mobile+instrumentation+audit-only sweep that drops the documentation check. Placed foundry first in the skills order (the entry-point door) ahead of the flagship it delivers, and made it the default command in the landing's hero terminal.

### 2026-07-11: document is idempotent, establish then reconcile

Reframed `document` from a create-once skill into an idempotent reconcile-to-a-fixed-point operation, meant to be run as often as wanted. Reasoning: the first run declares "I want documentation rigor" and establishes the surfaces (the Behind-the-Build hub, the deck); every run after should pull every documentation surface back up to date with what shipped, and change nothing already current. The design: detect state first (no hub -> establish; hub exists -> reconcile), and hold a convergence test, a second run with nothing changed between produces no diff. The reconcile pass walks the drift-prone surfaces (the hand-authored deck, the hardcoded showcase copy like terminal demos and skill counts, cross-surface agreement); the doc-set tabs render from source so they are inherently current. Positioned relative to `doc-drift`: doc-drift reconciles markdown-versus-code (implementation is truth); document reconciles the *presentation* of those docs (deck, hub, showcase copy) versus shipped capability, and leans on doc-drift for the doc-versus-code layer. Note: the manual "are document and foundry documented everywhere" coverage sweep is exactly what a document re-run should now perform automatically. Rejected: keeping it create-once with a separate "keep it in sync" nudge (that is what rots, the whole point is that a single rerunnable operation converges).

### 2026-07-12: foundry splits into check (read-only) and prepare (modifying)

Reworked `foundry` from a single conductor that ran all four gates in order (three of which modify the repo) into two explicit modes. Reasoning (from the product audit, issue 4): a readiness pass that silently runs `document`, `mobile`, and `instrumentation`, all of which change files, immediately before issuing a verdict is a build-and-audit pipeline masquerading as an assessment, and the same pass that changed the code should not also judge it. The split: `foundry check` (default) is read-only and reports which gates are met as a provisional scorecard; `foundry prepare` invokes the modifying skills to close the gaps; the authoritative verdict is an independent `/production-audit` run after prepare passes tests. Because assessment, modification, and judgment are three separate runs, before-and-after is measurable and the verdict is honest. Default is `check`, so bare "run foundry" never modifies; modification is opt-in via `prepare`. Reconciled the description across README, PRODUCT, CLAUDE, the deck, and the hero terminal (now `/foundry check`).

### 2026-07-12: Resolved document vs brand-voice with public/internal modes

`document` rendered the repo's docs (including PROMPTS/NOTES/FRICTION) into the public hub; `brand-voice` forbids rendering internal strategy, specs, or decision logs verbatim on a user-facing surface. The product audit (issue 2) flagged the contradiction, and the live site demonstrated it: the public hub rendered a mostly-empty FRICTION template and internal decision notes. The fix is not to abandon the hub but to name the audience with a mode. `document` now has three modes: `public` (curated, external-facing hub, only the deck/design-system/product/brand, no placeholders, no raw logs), `internal` (the raw source docs behind a gate, for continuity/inspection), and `reconcile` (the idempotent update pass). The prohibition is scoped to public surfaces; the gated internal portal may render raw docs, which is its job. `brand-voice` was updated to state the same scope from its side. On the site, the public `/behind` hub was curated down to Overview, Design system, Product, Brand; the Prompts, Notes, and Friction routes were deleted and those files removed from the doc whitelist, so the site cannot render a raw log or an empty template section. The raw docs still live in the repo (and, when the repo is public, on GitHub) for anyone who clones. Rejected: building a full authenticated internal portal on the site now, the repo itself is the internal source of truth for continuity; a gated on-site portal can come later if wanted.

### 2026-07-12: Install prioritizes npx, from-source is the fallback

The install surface leads with `npx skills@latest add horizon-foundry/foundry` (the generic `skills` CLI, github.com/mattpocock/skills), with clone + `make install` demoted to a "from source" alternative. Reasoning: a one-line npx command is the friendliest adoption path and matches how the skills ecosystem (and Impeccable) present installation. Verified against the `skills` CLI README before promoting it: it supports exactly this repo's layout (`skills/<name>/SKILL.md`) and installs skills as top-level `/commands` (not namespaced), so `/foundry` and `/production-audit` work directly. What the README does NOT state, and so is not asserted on the site: the exact uninstall/update commands and the Node version. That is why the from-source path is kept and documented in full, it is the one with exact `make uninstall` and `git pull && make install` lifecycle commands. Requirements copy avoids a fabricated Node version (just "Node"). The npx command cannot be end-to-end tested until the repo is public.

### 2026-07-12: Reports access is Supabase magic link (replaces HMAC gate + capability links)

Reports carry vulnerability findings, so `/reports` needs access control. The original control was a custom HMAC cookie + a single `REPORT_ACCESS_EMAILS` allowlist (whole-collection unlock, no per-report ownership, guessable). We designed a no-auth alternative (unguessable capability links + a local `/foundry list` keyring + hashed tokens), but it was machinery to work around not having identity. Decision (Craig): use Supabase magic link. Magic link proves email ownership (kills guessing) and collapses the design to "filter reports by `meta.owner`"; Craig already runs Supabase. Reports stay as JSON files (Supabase is used for AUTH ONLY, not storage), a signed-in user sees only their own reports, admin (via `ADMIN_EMAILS`) sees all, the public example stays ungated. This REPLACES both the HMAC gate and the capability-link plan. Full plan: `~/.claude/plans/foundry-reports-supabase-magic-link.md`. Sequencing: this lands BEFORE the public flip (Craig moved the publish after the auth). Rejected alternatives: keep the guessable email-only gate (insecure); capability links + keyring (too complex for the value).

### 2026-07-12: Magic-link implementation choices (token_hash landing, server-only env names, 404 over 403)

Three implementation decisions in the Supabase auth build that deviate from or sharpen the plan. (1) The magic-link landing route is `/auth/confirm` accepting `token_hash` + `type` and calling `verifyOtp` as the PRIMARY shape, with the plan's `?code=` + `exchangeCodeForSession` kept only as a fallback. Reasoning: current Supabase docs recommend the token_hash form for server-side email flows, and it is not PKCE-bound, so a link requested on a laptop still works when opened in the phone's mail client; the code shape fails cross-browser because the PKCE verifier cookie lives where the request was made. Requires pointing the Supabase Magic Link email template at `{{ .RedirectTo }}?token_hash={{ .TokenHash }}&type=email`; discovered at wiring time that the FREE tier cannot modify templates with the default email provider (and its built-in email allows only 2 sign-in emails per hour), so until custom SMTP or a paid plan lands, the code fallback keeps default emails working same-browser and cross-device links are a known limitation. (2) Env names are `SUPABASE_URL` / `SUPABASE_ANON_KEY`, NOT the conventional `NEXT_PUBLIC_*`: Next inlines `NEXT_PUBLIC_` vars at build time and the Fly Docker image builds without secrets, so runtime secrets under those names would silently never be read and the gate would fail closed forever. Possible only because the design has no browser Supabase client (sign-in is a server action, the session check is server-side). (3) Non-owner report slugs return 404, not 403, so the gate never confirms a report exists; `generateMetadata` resolves through the same access check so even a page title cannot leak. Also: sign-out is POST-only (a GET link would be triggered by prefetch), and there is no allowlist anywhere anymore; anyone can sign in and sees only what they own.

### 2026-07-12: PostHog instrumentation, two funnels, view events client-side

Wired PostHog per the instrumentation skill: the event plan was written into CLAUDE.md FIRST and the capture code conforms to it. Two funnels: suite adoption (pageview -> install_copied -> example_report_viewed; true activation happens inside Claude Code, out of the site's reach, so the example view is the honest value proxy) and reports access (magic_link_requested -> sign_in_completed -> report_viewed, the activation event). Identity: the Supabase user id is THE distinct id; the pre-auth intent event is keyed on the email and aliased to the user id at sign-in, and the gated layout identifies the browser, so client and server stages stitch into one person. Choices worth keeping: (1) view events (report_viewed, example_report_viewed) fire from a client component on mount, not in the server component, because App Router link prefetch renders server components and would inflate the funnel; (2) the browser key is a Docker build arg in fly.toml, not a Fly secret, because Next inlines NEXT_PUBLIC_* at build time (same reasoning as the Supabase env naming, opposite conclusion: this one MUST be NEXT_PUBLIC_ for posthog-js, and a phc key is public by design); (3) server capture is flushAt 1 / flushInterval 0 and never awaited (Fly machines auto-stop; send immediately or lose events); (4) sign-out resets the analytics identity client-side before the POST so shared browsers do not cross-attribute. Everything no-ops with no key. Constraint discovered: the available PostHog MCP key reaches only an unrelated org and lacks project/insight/dashboard write scopes, so the PostHog project and the saved funnel are created by Craig in the UI; the saved funnel to create is magic_link_requested -> sign_in_completed -> report_viewed. Rejected: server-side pageview capture (prefetch inflation, duplicates the SDK); capturing report content properties beyond the slug (reports are sensitive; the slug is enough to count).

### 2026-07-13: The impeccable pass, artifact-first hero, one-kicker cadence, CSP lesson

A dual-agent design audit/critique (impeccable) drove a six-action pass; the decisions worth keeping. (1) **The hero leads with the artifact, not theater.** The simulated typing terminal (the most saturated dev-tool hero of the era, and it could only replay canned output) was replaced by a report specimen: a real excerpt of the published example report rendered by the same components that render every report, with the refutation trust line. Deliberately shows a NON-perfect verdict (ship with known risks), which reads as credibility, not weakness. Rejected: keeping the terminal alongside (two competing hero objects) and a hand-drawn mock stamp (fake data on a trust surface). (2) **Section openings are a named cadence, not a reflex**: the tracked-caps kicker exactly once (hero), rule+title for sequence sections, bare display h2 for prose sections, and scroll entrances alternate rise/fade. The eyebrow-above-every-section grammar was the site's biggest AI tell. (3) **The report-excerpt exception** is codified in DESIGN.md: severity/verdict hues on marketing surfaces only inside elements framed as report excerpts; labels are luminance, never hue. (4) **The CSP lesson**: `connect-src 'self'` silently killed every client-side PostHog call in production for a product that markets instrumentation as a ship gate; found independently by three review agents. The rule is now written into next.config.ts: a new external service gets its exact hosts allowlisted in the same commit or the browser drops the traffic. (5) Severity chips painted a solid severity background under same-hue text (invisible labels on the credibility surface); the fix is a 12% tint + full-strength text, and the pattern to remember is never bg-X and text-X of the same token on one element. Two open design questions parked in TODOS: should the homepage BE the example report with marketing as margin annotations, and does the severity ramp need a colder mid-tone to keep amber-as-executable uncontested.

**Reversal (same day, Craig):** the specimen-led hero shipped and was pulled within hours. "Way too bold, in your face, and negative": a full-strength SHIP WITH KNOWN RISKS stamp as a stranger's first impression reads as a warning about the product, not evidence of honesty, and the terminal "was a better representation of the product, or at least more useful". The terminal returned to the hero (with the falsifiable /mobile output line softened and a faint "simulated run" label, the honest-boundary rule applied to our own theater), and the specimen moved to the worked-example section, replacing the bare stat grid there, where evidence sits next to the story of the run. The lesson recorded: the critique's "lead with the artifact" logic optimized for distinctiveness, but the hero's job is representation; an artifact whose headline is a caution belongs beside its context, not above the fold. The homepage-as-report open question stays parked, this reversal is a data point against it.

### 2026-07-13: Narrow the claim at the flip (positioning decision)

A second external adversarial critique scored Foundry 8/10 as an AI-assisted delivery system and 5.5/10 as an end-to-end product-building system, and charged that "a suite of good product-building practices" overclaims: the suite has no discovery/framing, operability, release-strategy, or post-launch-learning disciplines, so the claim asserts rigor the portfolio does not carry. Decision (Craig): narrow the claim NOW, before the public flip. The positioning becomes "AI-assisted product delivery and release-readiness practices"; the line is "Foundry makes AI-assisted product delivery honest, continuous, and release-ready." Reasoning: the suite's entire brand is honesty (the honest boundary, evidence classes, refutation); an overclaimed category line is the one place the brand contradicts itself, and it is the first thing a skeptical evaluator will test. The broader claim can return when Phase 14's lifecycle work (frame, policy-driven gates, operability, post-launch learning) actually ships. Also folded into the pre-flip pass: the verification-rule contradiction (public copy promises refutation of every verdict-driving finding; the skill required only Critical+High) and verdict-scope honesty (justifications open by naming assessed scope). Rejected: renaming shipped skills per the critique (/handoff, /mobile-web-readiness), recognition beats taxonomy; deleting the deck/hub conventions, they are house style to label, not defects.

### 2026-07-13: frame is an auditor, not an interviewer (mechanism decision)

frame's first draft ran a guided interview: the skill asked nine discovery questions and wrote the answers into a product brief. Reworked the same day, before merging, to a declare-and-audit shape: the builder authors the one-page frame (or asks for a draft assembled only from materials that already exist, every entry marked as an unconfirmed assumption), and the skill audits the declaration the way the suite audits code: blank or vague entries are findings, unlabeled assumptions are findings, and nothing enters the frame the builder did not state or confirm. Reasoning: elicitation mechanics (generated question flows, guided discovery, decomposition into sub-problems, readiness-scored specs) are the mechanism space of dedicated discovery products, and Foundry should not be a lesser copy of that category; the suite's identity everywhere else is evaluative (verdicts, evidence classes, empty-cells-are-findings), so the frame skill is strongest speaking that same language. The constraint is suite-wide and binding: Foundry audits declarations, it does not elicit. It also renames the artifact vocabulary away from "brief" (a discovery-product noun) to "the frame". Applied in the same pass: scaffold's "short interview" became "a few setup answers the builder supplies", and 12.9's brand-voice fix specifies human approval of a declared identity rather than an interview. Rejected: keeping the interview with different question wording (same mechanism, cosmetic change) and dropping the skill entirely (the gap both external critiques named is real; the evaluative shape fills it distinctively).

### 2026-07-13: Consolidation, 11 skills to 9 (justify-or-cut before the flip)

Craig challenged whether every skill earns its slot ("we don't want to publish a bunch of weak concepts that don't really do much"), starting with resume. A justify-or-cut pass (grounded in the day's six-reviewer audit + two critiques) asked not "does it work" but "does it deserve to exist", and merged what overlapped. Outcomes: (1) **resume CUT.** It produced no artifact, was redundant with the plan chain + a CLAUDE.md instruction + a session-start hook, and the suite did not even dogfood it as a skill (this repo resumes off the hook and TODOS, never by invoking /resume). Its "read the indexed plan before acting" rule is now a standing instruction scaffold seeds into a project's CLAUDE.md, framed honestly as an invariant, not a judgment call (the second critique's skills-vs-hooks point). (2) **doc-drift MERGED into document.** Both kept the project's words honest; document now owns two layers, truth (docs vs code) and presentation (deck, hub, showcase copy). The merge also applied the audit's P0 fix: the authority model is no longer "implementation is truth" flat, it classifies deterministic mismatches (repair the doc) from suspected regressions (a failing test or an intent claim means flag it, never document the bug as the contract), and the filesystem-mirror registry rule is now conditional. (3) **brand-voice KEPT**, Craig's call: DESIGN.md and BRAND.md are two distinct artifacts (looks vs voice), and brand-voice is what makes BRAND.md load-bearing; folding it into document would lose the audience distinction (maintainer docs vs outward product voice). The suite is now a clean lifecycle arc: frame to scaffold to (document, instrument, mobile, phase-plan) to foundry to production-audit, every skill distinct, none filler. Rejected (from the critique): renaming shipped skills, and splitting document into /docs + /showcase (the two layers are one job in one place). All marketing surfaces (README, PRODUCT, CLAUDE tree + overview, the Behind-the-Build deck, lib/skills.ts copy) and both reference docs reconciled in the same change; retired skill routes 404.

### 2026-07-13: Phase 12.9 hardening decisions (positioning, gates, ownership, enforcement)

Four decisions from the pre-public hardening pass, recorded together because they share one driver: the suite's brand is honesty, so every place it overclaimed or hardcoded a preference had to move before the flip.

1. **Positioning is "software delivery integrity", five promises.** Supersedes the interim "AI-assisted product delivery" line from the 2026-07-13 critique entry above. Reasoning: the five promises (intent declared and audited; execution context preserved; outcome instrumented; docs aligned with reality; readiness audited) map one-to-one onto shipped skills, so the claim is checkable the same way the audit's matrices are: a promise with no skill behind it would be a blank cell. Rejected: keeping the broader product-building claim until Phase 14 ships post-launch learning; an overclaimed category line is the first thing a skeptical evaluator tests.
2. **Ship gates went from universal law to policy-resolved.** "You do not ship without mobile + instrumentation" is now "the release policy names the gates this launch requires; skipping one silently is what is forbidden." This deliberately softens the earlier explicit stance (decided in the Phase 12.9 kickoff): a CLI has no mobile surface and a privacy-sensitive tool must not phone home, so the universal form was false at the edges, and a rule that is sometimes false teaches readers to ignore it. The invariant that survives is honesty (not-applicable and waived are stated, never silent).
3. **init/list moved from foundry to production-audit.** Report attribution and history are report concerns, owned by the skill that emits reports; foundry is purely check/prepare/handoff. Rejected: a third identity skill (too much surface for two tiny modes).
4. **Invariants moved from prose to machine checks.** scripts/validate-report-invariants.mjs enforces the verification rule, scope honesty, stats accuracy, ID-prefix/dimension agreement, no em dashes, and no personal paths in make validate; skill-authoring.md now says plainly that an opening gate improves compliance but does not guarantee it. Reasoning: the suite tells adopters to sort judgment calls from invariants; it has to practice that on its own contract.

### 2026-07-13: Dogfood fold-back scope (fix reconciliation, backlog code)

The Phase 12.9 dogfood (frame + full self-audit) ended with a scoping decision: only reconciliation-class findings landed on the branch (the CHANGELOG nine-skill fix, the em-dash sweep), while every code-touching remediation (CI, gate unit tests, Supabase timeouts, emailRedirectTo pin, generateStaticParams, a11y lows) went to a tagged TODOS backlog section for an explicit pre-flip vs post-flip call. Reasoning: the PR should ship the exact snapshot the audit assessed; mixing unaudited new code into it would both invalidate the report's auditedCommit and delay the flip-blocking merge. Rejected: fixing the quick wins inline (tempting, each is small, but the sum is a new unaudited surface). Two supporting observations recorded for the method: the two refuted highs (SDK-rejection claims killed by reading vendored @supabase/auth-js source) are live proof the adversarial verification phase earns its cost, and frame's first real run finding that the framing suite had no frame of its own is the kind of self-application result the dogfood step exists to surface. The frame's entries 6 and 9 stay blank pending Craig; blanks are findings, not gaps to invent answers for.

### 2026-07-14: Positioning copy, tune-vs-fixed and outer-loop (Phase 12.10)

A colleague evaluating the suite asked two things that turned out to be one concern: "can I tune what is production ready, or is it based on your own recipe?" and "have you seen Matt Pocock's work?" Both ask whether Foundry is a personal idiolect or connects to the shared practice. Decision: answer both on the public surfaces, in the same phase, as copy only.

Two framings, now canonical in PRODUCT.md and echoed on every surface:

1. **What you tune vs what's fixed.** You set the scope (release policy: which gates apply, risk tier, audience, accepted risks on the record; audit dimensions resolve by applicability). You do NOT set the bar (severity rubric, evidence classes, verdict meaning are fixed). Reasoning: a standard you can lower until everything reads green is the self-serving "looks fine to me" the audit exists to replace. The product is opinionated where opinion IS the product. This owns the "your own recipe" objection instead of conceding it, which is stronger than pretending infinite configurability. Rejected: making the severity bar tunable (guts the trust); a `risk_tier`-moves-the-verdict-tolerance extension was deliberately left as an OPEN post-launch decision, not built (it is a real change to how the flagship reasons, Craig's call, better decided once strangers actually use the policy).
